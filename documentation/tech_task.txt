The goal is to build a tabular learing framework/algo for phd thesis (that is later will be appied to benchmark datasets for research). The idea is to eliminate "expert" feature engineering in tabular data machine learning. One passes tabular data "as is" to the framework and gets trained algo for prediction. The idea of the framework is to use deeplearning for feature engineer with deeplearning (via producing embeddings) and then feeding it to the "classical" tabular data machine learining algos. Thus we keep deep learning felixibility and convinience and get the great results of "classical" models like XGBoost for tabular data, and all of it with no "expert" feature engineering step.

# Technical Specification (Developer Handoff)

## Hybrid Tabular Framework: PyTorch Embeddings → Classical Predictor (Embeddings-Only)

### 0. Goal

Implement a Python framework that, given a tabular dataset and configuration, automatically:

1. infers schema (numeric/categorical)
2. trains/tunes a PyTorch tabular embedding model (from scratch + optionally via PyTorch tabular libraries)
3. extracts embeddings for all samples
4. tunes/trains a classical predictor on embeddings only
5. evaluates and logs results reproducibly across benchmark datasets (classification + regression)

**Hard constraints**

* Deep learning: **PyTorch only**
* **No TensorFlow/Keras**
* **No manual feature engineering** (no handcrafted features; no expert steps)
* Classical models must use **embeddings only** as input
* Tasks: **binary classification, multiclass classification, regression**
* Hyperparameter tuning: **Hyperopt** (not Optuna, not Ray Tune)
* Experiment tracking: **no MLflow** (local JSON/CSV/Parquet logging only)

---

## 1. Key Design Principle: “Embeddings-only”

The framework must treat the tabular dataset as “raw” inputs and perform only:

* minimal **schema-driven normalization** needed for stable training (allowed)
* categorical **index mapping** for embedding tables (allowed)
* missing-value handling (allowed, but must be non-expert, generic)

**Forbidden**

* One-hot encoding, target encoding, count encoding
* domain feature construction
* binning, polynomial features, interactions
* feature selection
* any custom dataset-specific engineering

---

## 2. Pipeline Overview

**Input**: `X (DataFrame), y (Series/array), task_config, budget_config`

**Output**: `RunResult` + local artifacts folder

### 2.1 Stages

1. **Data profiling + schema inference**
2. **Split generation** (CV or train/val/test)
3. **Minimal preprocessing**
4. **Embedding model selection + HPO** (Hyperopt)
5. **Embedding extraction**
6. **Classical model selection + HPO** (Hyperopt)
7. **Final evaluation + aggregation**
8. **Local logging + artifacts**

---

## 3. Supported Tasks (Only)

* Binary classification
* Multiclass classification
* Regression

No ranking.

---

## 4. Minimal Preprocessing Rules (Generic, Non-expert)

### 4.1 Missing values

* Numeric: fill with train-fold median + add **no** indicator feature (indicator would be “engineering”; disallowed)
* Categorical: map missing to `"__MISSING__"` category id

### 4.2 Numeric scaling (allowed)

* StandardScaler fitted on train fold only
* Optional: robust scaler (config toggle)

### 4.3 Categorical mapping (allowed)

* Per categorical column: fit vocabulary on train fold
* Unseen in val/test → `"__OOV__"`

**Important**: Classical models never see categoricals directly. They only see learned embeddings output vectors.

---

## 5. Embedding Stage (Deep Learning, PyTorch)

### 5.1 Requirements

Embedding stage must output a **fixed-length dense vector** per sample:

* `E ∈ R^(N × d)`
* Used as the *only* input to classical models.

### 5.2 Two Embedding Sources

#### Mode A — From-scratch PyTorch architectures (required)

Implement at least:

1. **Tabular MLP with categorical embeddings**

   * Learn embedding tables per categorical feature
   * Numeric projection layer
   * Concatenate → MLP trunk
   * Output embedding vector from penultimate layer

2. **FT-Transformer-style model** (strongly recommended)

   * Feature tokenization (numeric as learned linear token; categorical as embedding token)
   * Transformer encoder
   * CLS pooled output = embedding

Developers should implement these in native PyTorch.

#### Mode B — Pre-existing PyTorch tabular embedding frameworks (optional but supported)

Adapters (PyTorch-only):

* `pytorch-tabular` (if acceptable in your environment)
* `rtdl` / “Revisiting Deep Learning Models for Tabular Data” implementations
* PyTorch TabNet implementation (only if fully PyTorch)

**Adapter rule**: All library models must expose:

* `fit(...)`
* `encode(X) -> np.ndarray`

---

## 6. Hyperparameter Optimization (Hyperopt)

### 6.1 General strategy

Use **Hyperopt TPE** with early stopping and trial pruning by:

* limiting max epochs and using validation-based early stop
* enforcing time budget per stage

### 6.2 Objective for embedding search (critical design choice)

Because embeddings are used downstream, the embedding HPO objective is:

**Primary objective** = validation score of a **proxy classical model trained on embeddings**.

Per trial:

1. Train embedding model on train fold (early stop on val)
2. Extract embeddings `E_train, E_val`
3. Train a cheap proxy classical model on `E_train`
4. Score on `E_val`
5. Return negative score (Hyperopt minimizes)

**Proxy model**

* Classification: LogisticRegression (or linear SVM)
* Regression: Ridge regression

Reason: fast + stable; enables many trials.

### 6.3 Search space: From-scratch MLP (minimum)

Hyperopt should tune:

* embedding dims policy (global dim or per-column rule)
* MLP depth (2–8)
* hidden width (128–2048)
* dropout (0–0.5)
* activation (ReLU/GELU/SiLU)
* normalization (LayerNorm on/off)
* lr (log-uniform)
* weight_decay (log-uniform)
* batch size (categorical choices)
* optimizer = AdamW (default)

### 6.4 Search space: FT-Transformer (recommended)

* n_layers (2–6)
* d_model (64–512)
* n_heads (4–16)
* attn_dropout, ff_dropout
* lr, weight_decay, batch size

## 7. Classical Model Stage (Embeddings-only)

### 7.1 Allowed classical model families

All consume `E` only.

**Classification**

* LogisticRegression
* LinearSVM (optional)
* RandomForestClassifier
* ExtraTreesClassifier
* XGBoostClassifier
* LightGBMClassifier
* CatBoostClassifier

**Regression**

* Ridge
* ElasticNet (optional)
* RandomForestRegressor
* ExtraTreesRegressor
* XGBoostRegressor
* LightGBMRegressor
* CatBoostRegressor

### 7.2 Classical model tuning (Hyperopt)

Use Hyperopt for:

* model family selection (categorical choice)
* hyperparams per family

Implementation:

* define `space = hp.choice("model_type", [...])`
* each branch has its param space
* objective = validation metric on embeddings

Early stopping:

* LightGBM/XGBoost: use built-in early stopping with validation set

**Output**

* `best_classical_model` + config
* retrain on train+val embeddings
* evaluate on test

## 8. Evaluation Protocol

### 8.1 Splits

Support:

* K-fold CV (default: 5)
* train/val/test single split (optional)

### 8.2 Metrics

**Binary classification**

* Primary: ROC-AUC
* Secondary: logloss, PR-AUC (if imbalanced)

**Multiclass**

* Primary: accuracy (or macro-F1)
* Secondary: logloss

**Regression**

* Primary: RMSE
* Secondary: MAE, R²

### 8.3 Reporting

For CV:

* mean ± std of each metric
* rank vs baselines (optional but useful)

### Unit Testing Guidelines

#### Framework and Core Principles

This project uses **pytest** as the testing framework. All unit tests must follow these strict guidelines to ensure consistency, clarity, and maintainability.

### Forbidden Practices

#### 1. No Fixtures
- **NEVER** use pytest fixtures
- Fixtures are strictly forbidden in this codebase
- All test data must be generated explicitly within each test function

#### 2. Avoid Mock Functionality whenever possible

### Data Generation Requirements

#### 3. Explicit Random Data Generation
- When a tested function needs data to work with, generate random data **explicitly** in the unit test module
- Use libraries like `random`, `numpy`, `pandas` to create test data
- Make data generation visible and understandable within the test
- Fix random seeds for reproducibility

### Demo-Purpose Testing

#### 4. Educational Unit Tests
Unit tests serve dual purposes:
- **Testing functionality**: Verify that functions work correctly
- **Demo/Documentation**: Serve as live examples of how functions work

Tests should be structured so that anyone can:
1. Open an ipython console
2. Run test code piece by piece (imports + helper functions + test body)
3. Understand function inputs, outputs, and behavior

### Debugging and Visibility

#### 5. Use Print Statements for Visibility
- Include `print()` statements in unit tests to show:
  - What the test is doing
  - Preliminary results
  - Intermediate values
- Prints help with debugging and understanding test flow
- **Important**: Prints are logs for visibility, NOT substitutes for `assert` statements
- All tests must still use proper `assert` statements for validation

### Test Structure Example

```python
# imports always go ath the head of the module
import random
from some_module import some_function

# Fix random seed for reproducibility
random.seed(42)

# Function to generate test data explicitly
def generate_random_list_of_ints(n: int, random_seed: int = None):
    if random_seed is not None:
        random.seed(random_seed)
    return [random.randint(1, 100) for _ in range(n)]

def test_some_function():
    test_data = generate_random_list_of_ints(n=10, random_seed=42)
    expected_result = sum(test_data)
    
    print(f"Testing with data: {test_data}")
    print(f"Expected sum: {expected_result}")
    
    # Call the function
    result = some_function(test_data)
    print(f"Actual result: {result}")
    
    # Assert the result
    assert result == expected_result, f"Expected {expected_result}, got {result}"
    print("Test passed!")
```

### Key Reminders

- Every test should be self-contained and runnable independently
- Generate all necessary data within the test function
- Use descriptive print statements to document test progress; this does not substitute proper assert statements as in proper unit testing
- Always validate results with proper assert statements
- Make tests educational - they should teach how the function works
